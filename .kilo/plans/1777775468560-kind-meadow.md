# Bugfix Plan: MCP receive/send reliability

## Context

During a simple "receive pending → send → receive" flow across two OWS wallets, we hit 4 distinct issues that caused 6+ retry attempts for what should be a 3-step operation. Root causes traced to `src/mcp.ts`, `src/nano-actions.ts`, and `src/nano-actions.ts` timeout defaults.

---

## Bug 1 (Critical): `readersFor()` ignores `config_set` RPC URL

**File**: `src/mcp.ts:121`

**Root cause**: Every MCP handler calls `readersFor(String(args?.rpcUrl ?? ''))`. When the user doesn't pass `rpcUrl` in the tool call (the normal case), this becomes `readersFor('')`. Inside `getNanoClient`:

```typescript
const rpc = (explicitRpc || state.config.rpcUrl || process.env.NANO_RPC_URL || '').split(',').filter(Boolean);
```

`''` is falsy, so it falls through to `state.config.rpcUrl` — this part is correct. However, the `nanoClient` caching check:

```typescript
if (state.nanoClient && !explicitRpc && !explicitWork) {
  return state.nanoClient;  // Returns STALE cached client
}
```

`!''` is `true`, so after the first tool call creates and caches a client, ALL subsequent calls reuse it — even after `config_set` sets `state.nanoClient = undefined`. The issue: `config_set` does clear the cache, but the very next call to `getNanoClient('')` creates a fresh client with the correct config... and then caches it. This should work.

**The real problem**: After investigation, the caching logic is actually correct. The real issue is that the MCP server is a long-running process. When default endpoints (rainstorm, nanoslo) get rate-limited, the `EndpointPool` inside `@openrai/nano-core` puts them on exponential backoff cooldown (500ms × 2^(failures-1), capped at 30s). `config_set` resets `state.nanoClient` but the NEW client still uses the same default endpoints — they're just on cooldown in the new pool too.

**Fix**: Change `readersFor()` to always pass `state.config.rpcUrl` as the default when no explicit `rpcUrl` is provided, instead of relying on the empty-string fallback. More importantly, when `config_set` is called with a NEW rpcUrl, the new NanoClient should be created with ONLY that URL (not the defaults).

```typescript
// mcp.ts — change readersFor to resolve rpcUrl properly
function readersFor(explicitRpcUrl?: string): NanoReaders {
  const effectiveRpc = explicitRpcUrl || state.config.rpcUrl || '';
  const client = getNanoClient(effectiveRpc || undefined);
  // ...
}
```

And in every handler, change from:
```typescript
readersFor(String(args?.rpcUrl ?? ''))
```
to:
```typescript
readersFor(args?.rpcUrl ? String(args.rpcUrl) : undefined)
```

This way, `undefined` means "use config/default", `''` means "use config/default", and an actual URL means "use this URL". The `getNanoClient` function already handles this correctly with `explicitRpc || state.config.rpcUrl || ...`.

**Test**: Add a test that calls `config_set` with a custom rpcUrl, then calls `receive` and verifies the NanoClient was created with the custom URL (not the defaults).

## Bug 2 (Critical): Open block balance sums ALL pending but link references only FIRST

**File**: `src/nano-actions.ts:424-436`

**Root cause**: When an unopened account has multiple pending blocks:

```typescript
const amountRaw = pending.reduce((sum, item) => sum + BigInt(item.amount), 0n).toString();
const newBalance = (currentBalance + BigInt(amountRaw)).toString();
// ...
link: pending[0].hash,  // Only FIRST pending block's hash
```

The `balance` field reflects the sum of ALL pending amounts, but `link` only references the first pending block. The Nano node validates that `balance_delta == amount_of_linked_send`. When they don't match, it rejects with "Balance and amount delta do not match".

**Our workaround during the session**: Using `--hash` to filter to a single pending block worked because then `amountRaw` = that one block's amount, and `link` = that block's hash. They matched.

**Fix**: Process ONE pending block at a time. For the first block on an unopened account, build an `open` block. For subsequent blocks, fetch the updated account info (new frontier) and build `receive` blocks.

```typescript
// nano-actions.ts — executeReceive rewrite
export async function executeReceive(...): Promise<ReceiveResult> {
  // ... resolve account, fetch info, fetch receivable ...
  
  const pending = options.onlyHash 
    ? receivable.filter((item) => item.hash === options.onlyHash) 
    : receivable;
  if (!pending.length) { /* return early */ }

  const received: Array<{ hash: string; amountRaw: string }> = [];
  let currentFrontier = opened ? (info as AccountInfoResponse).frontier : ZERO_32_HEX;
  let currentBalance = opened ? BigInt((info as AccountInfoResponse).balance) : 0n;
  let isOpened = opened;

  for (const item of pending) {
    const representative = isOpened 
      ? (info as AccountInfoResponse).representative 
      : requireRepresentativeAddress(ctx.config, options.representative);
    
    const newBalance = (currentBalance + BigInt(item.amount)).toString();
    const subtype = isOpened ? 'receive' : 'open';
    
    const blockInput: StateBlockHashInput = {
      accountPublicKey: decodeNanoAddress(account.address).publicKey,
      previous: currentFrontier,
      representativePublicKey: decodeNanoAddress(representative).publicKey,
      balanceRaw: newBalance,
      link: item.hash,
    };

    const submitted = await signWorkAndProcess(walletName, account.chainId, blockInput, subtype, index, readers);
    
    received.push({ hash: submitted.txHash, amountRaw: item.amount });
    currentFrontier = submitted.txHash; // New frontier after this block
    currentBalance = BigInt(newBalance);
    isOpened = true; // After first block, account is opened
    
    ctx.appendTransaction?.({ /* ... */ });
  }

  return {
    address: account.address,
    received,
    balanceRaw: currentBalance.toString(),
    balanceXno: rawToNano(currentBalance.toString()),
  };
}
```

**Test**: Add a unit test with a mock account that has 3 pending blocks. Verify that 3 separate receive blocks are built, each with the correct `link` and `balance` fields.

## Bug 3 (Medium): Default timeout too short for public node PoW

**File**: `src/nano-actions.ts:12`

**Root cause**: `DEFAULT_TIMEOUT_MS = 15000` (15 seconds). This timeout wraps the entire RPC call including PoW generation. Public nodes often take 10-20 seconds for `work_generate`. The CLI `send` failed with "Timed out after 10000ms" (the MCP server uses a different default).

**Fix**: Separate the PoW timeout from the general RPC timeout. PoW operations should have a longer timeout (e.g., 60s) while general RPC reads keep the shorter timeout (15s).

In `signWorkAndProcess`, the `workGenerate` call should use a PoW-specific timeout:

```typescript
// nano-actions.ts — signWorkAndProcess
const powTimeoutMs = Math.max(readers.powTimeoutMs ?? 60_000, 30_000);
work = await withTimeout(
  readers.workGenerate(workRoot, difficulty),
  powTimeoutMs,
  `PoW generation timed out after ${powTimeoutMs}ms`
);
```

Add `powTimeoutMs` to the `NanoReaders` type and set it in `readersFor()`.

In `mcp.ts`:
```typescript
function readersFor(...): NanoReaders {
  return {
    // ... existing readers ...
    powTimeoutMs: state.config.timeoutMs ? state.config.timeoutMs * 4 : 60_000,
  };
}
```

**Test**: Verify that the PoW timeout is independent of the RPC timeout.

## Implementation order

1. **Bug 2** (open block balance) — highest impact, causes the most confusing errors
2. **Bug 1** (readersFor config_set) — second highest impact, causes "All endpoints exhausted"  
3. **Bug 3** (timeout) — medium impact, causes intermittent PoW failures
4. **Bug 4** (docs) — low impact, improves troubleshooting
